Parte 1

O que significa "treinar um agente"

Quando alguém diz "vou treinar meu agente", três coisas diferentes podem estar sendo ditas. Confundir é caro.

A) Fine-tuning de verdade (ajustar pesos)

Modificar a rede neural do modelo. Exige dataset gigante, meses de engenharia, custa centenas de milhares de dólares. Ninguém da nossa escala faz isso. Nem a OpenAI recomenda pra maioria dos casos — ela diz expressamente: "tente RAG antes".

B) System prompt + few-shot examples

Escrever uma instrução boa ("você é um especialista em copy do programa Elo, tom assim, evita isso, sempre inclua aquilo") e colar 2–5 exemplos de saída ideal. É o que já temos — seus 60 prompts do Pinguim.

C) RAG — Retrieval-Augmented Generation

Em vez de enfiar todo o Cérebro no prompt (inviável), o agente busca na hora só os trechos relevantes e usa esses. É o que vamos construir agora em cima do seu Supabase.

Resumo prático

"Treinar o agente de copy do Elo" = System prompt do Copy Chief (que você já tem) + RAG no Cérebro Elo (que vamos fazer). Fine-tuning fica fora. A qualidade não perde. O custo cai 20–50x.

Parte 2

RAG é o coração da redução de token

Vamos no número pra doer. Imagine o Cérebro Elo quando tiver 100 fontes grandes (aulas transcritas, páginas, depoimentos):

Abordagem ingênua

Jogar o Cérebro inteiro no prompt.

~400 mil tokens por chamada.

~US$ 1 por requisição. Inviável.

Abordagem RAG

Buscar só os 8 trechos mais relevantes e injetar no prompt.

~8 mil tokens por chamada.

~US$ 0,02 por requisição. Escalável.

Como o RAG funciona no nosso sistema

1. Quebro cada fonte em pedaços de ~500 tokens (chunks) └── 1 aula de 2h vira ~40 chunks 2. Passo cada chunk pela OpenAI text-embedding-3-small └── Ele vira um vetor de 1536 números (embedding) └── Custo: ~US$ 0,00002 por chunk. Irrelevante. 3. Guardo os vetores no Supabase └── Extensão pgvector nativa — zero serviço extra 4. Quando agente precisar escrever algo: ├── Transforma a pergunta em vetor também ├── Pergunta pro Supabase: "dá os 8 chunks mais próximos" └── SÓ esses 8 chunks vão no prompt do GPT

Parte 3

Formato das fontes no Supabase

Duas tabelas — uma pro humano (UI do Mission Control), outra pro agente (consulta RAG).

Tabela 1 — cerebro_fontes (o que o humano vê)

cerebro_fontes (
  id            uuid primary key,
  cerebro_id    uuid references cerebros,
  tipo          text,          -- aula, página, depoimento, etc (dinâmico, livre)
  origem        text,          -- discord, whatsapp, upload, expert, scrap…
  titulo        text,
  conteudo_md   text,          -- markdown normalizado (ver "por quê" abaixo)
  autor         text,
  url           text,
  metadata      jsonb,         -- duração, tags, data-gravação, qualquer extra
  criado_em     timestamptz
)

Tabela 2 — cerebro_fontes_chunks (o que o agente lê)

cerebro_fontes_chunks (
  id              uuid primary key,
  fonte_id        uuid references cerebro_fontes on delete cascade,
  cerebro_id      uuid,         -- denormalizado pra filtrar rápido
  chunk_index     int,          -- posição dentro da fonte
  conteudo        text,         -- pedaço de ~500 tokens
  embedding       vector(1536), -- pgvector — a "assinatura" semântica
  token_count     int,
  criado_em       timestamptz
)

Por que essa estrutura é certa

  • Markdown normalizado como único formato do banco
    PDF, DOCX, TXT, CSV — tudo é convertido pra MD antes de entrar. Agente nunca vê formato original, só texto limpo. Um único parser de contexto em vez de cinco.
  • metadata jsonb para extensibilidade sem mexer em schema
    Tipo novo aparece (ex: "entrevista", "podcast")? Só jogar em tipo e detalhes em metadata. Zero migration.
  • Chunks separados da fonte
    Uma aula aparece 1 vez no Kanban (UI lê cerebro_fontes), mas tem 40 chunks pro agente (RAG lê cerebro_fontes_chunks). Apagou a fonte? on delete cascade limpa os chunks.
  • pgvector dentro do Postgres
    Busca semântica nativa. Zero Pinecone, zero Weaviate, zero serviço extra pra pagar. Se o Supabase tá de pé, RAG tá de pé.
Parte 4

As 3 portas de entrada de fontes

Pergunta que sempre aparece: "quem transforma o arquivo em vetor? Meu computador? O navegador? O Supabase?". Resposta: depende de qual das 3 portas está sendo usada. Mas o destino é sempre o mesmo — tabela cerebro_fontes_chunks no Supabase.

Em todos os 3 casos, o vetor (embedding) é gerado chamando a API text-embedding-3-small da OpenAI. O que muda é quem faz a chamada. O custo é irrelevante: cada chunk custa US$ 0,00002. Uma aula de 2h inteira custa ~R$ 0,005 pra vetorizar.

Porta 1 — Carga inicial (uma vez só, do que já existe)

Hoje já temos material: transcrições no VS Code, PDFs, briefings, planilhas. Isso entra via script rodando na máquina do André, uma vez só, usando a chave OpenAI dele (que fica local, nunca vai pro GitHub).

Arquivos em c:\Squad\cerebros\elo\*.md ↓ Script node scripts/import-elo-local.mjs (roda local) ├── lê os .md ├── chunka em pedaços de ~500 tokens ├── chama OpenAI embeddings (com OPENAI_API_KEY do .env local) └── INSERT direto no Supabase (com SERVICE_ROLE_KEY) ↓ Cérebro Elo populado — rodou 1x, fim.

Por que local e não Edge Function? Porque esse material não passa por internet pública — tá no disco, só faz sentido processar direto dali. Além disso, carga inicial pode ser grande (1000+ chunks), e rodar uma vez na sua máquina é mais simples que subir tudo pro Supabase Storage só pra processar.

Porta 2 — Upload manual pelo painel

Alguém da equipe (você, sua sócia, um analista) abre o Mission Control, clica + Alimentar, arrasta um PDF. Quem gera o vetor? Uma Edge Function do Supabase — não o navegador.

Browser faz upload → Supabase Storage ↓ Supabase dispara Edge Function "ingest-fonte" ├── pega o arquivo do Storage ├── extrai texto (PDF/DOCX → Markdown) ├── chunka em ~500 tokens ├── chama OpenAI embeddings (chave fica SECRETA no servidor) └── INSERT em cerebro_fontes + cerebro_fontes_chunks ↓ Painel atualiza: "Fonte processada · X chunks"

Por que não chamar OpenAI direto do navegador?

Porque qualquer um com F12 no Chrome veria a chave e roubaria. Chave de API NUNCA vai pro front-end. Fica como secret no Supabase, só a Edge Function enxerga. Edge Function é gratuita até 500 mil execuções/mês — não vamos chegar perto.

Porta 3 — Automação (crons)

Você configura pelo painel: "a cada hora, puxar mensagens novas do canal #depoimentos do Discord e virar fonte no Cérebro Elo". Quem roda? A mesma Edge Function da Porta 2, mas disparada automaticamente por pg_cron (cron nativo do Postgres que vem no Supabase).

pg_cron dispara a cada 1 hora (configurado pelo painel) ↓ Edge Function "cron-discord-depoimentos" ├── conecta no Discord com bot token (secret) ├── pega mensagens desde a última execução ├── classifica tipo via GPT-4o-mini (depoimento? sacada?) ├── chunka + embeda └── INSERT em cerebro_fontes + cerebro_fontes_chunks ↓ Próxima execução só vai pegar mensagens ainda mais novas

Comparação rápida

Porta Quem dispara Onde roda Frequência
1. Carga inicial André via terminal Máquina do André 1 vez só, por produto
2. Upload manual Equipe no painel Edge Function Supabase Sob demanda
3. Cron automático pg_cron agendado Edge Function Supabase Recorrente (ex: 1h, 6h, diário)

Fim da história: não importa de qual porta veio — a fonte acaba no mesmo lugar (Supabase), no mesmo formato (tabela + chunks vetorizados). A partir daí, qualquer agente consulta igual, sem saber de onde veio.

Bônus: 2 APIs OpenAI diferentes no sistema

Confusão comum: a OpenAI é chamada em dois momentos diferentes, com APIs diferentes. Vale separar pra não confundir custo:

Na ALIMENTAÇÃO

text-embedding-3-small

Transforma texto em vetor. Usado 1 vez por chunk, fica guardado. Custo: US$ 0,00002 por chunk.

No CONSUMO

gpt-4o / gpt-4o-mini

Gera resposta (página, copy, análise). Usado toda vez que um agente é acionado. Custo: centavos por tarefa.

Parte 5

Fluxo de ponta a ponta

Alimentação (quando o André sobe 1 arquivo no painel)

Usuário sobe "Aula 4.pdf" no Cérebro Elo ↓ Edge Function "extract-text" ├── lê PDF e extrai texto └── converte pra Markdown limpo ↓ INSERT em cerebro_fontes (1 linha) ↓ Trigger do Postgres chama Edge Function "chunk-embed" ├── quebra o MD em chunks de ~500 tokens ├── chama OpenAI text-embedding-3-small └── INSERT em cerebro_fontes_chunks (N linhas) ↓ Mission Control (UI) atualiza: ├── +1 coluna "Aulas" no Kanban (se primeira) ├── +1 card dentro da coluna └── Persona Elo flagged "precisa re-sintetizar"

Consumo (quando o agente de copy Elo escreve uma página)

Usuário: "Escreve página de vendas focada em consultores" ↓ Agente Copy-Elo monta request RAG: ├── transforma a pergunta em vetor └── query no Supabase: "top 8 chunks do Cérebro Elo mais próximos desse vetor" ↓ Prompt final pra OpenAI: [System prompt do Copy Chief — cacheado] [8 chunks do Cérebro Elo] [Pergunta do usuário] ↓ OpenAI retorna página ↓ Agente retorna ao usuário + salva registro da execução (audit trail pra Qualidade)

Observação: não existe "re-treinamento" quando o Cérebro muda. Cada request busca o estado atual do banco. Adicionou fonte agora? Próxima chamada já usa. Cérebro = estado vivo; agente = função pura que consulta.

Parte 6

Como o Cérebro alimenta tudo automaticamente

Você perguntou: "se eu mexer no Cérebro, o agente precisa ser re-alimentado?". A resposta é não. Mas existem algumas rotinas automáticas que rodam em cima:

GatilhoRotina automáticaOnde roda
Fonte adicionada Chunk + embed Edge Function (Supabase)
Fonte editada Re-chunk + re-embed só daquela Edge Function (Supabase)
Fonte removida Cascade limpa chunks Postgres (trigger)
Cérebro atualizado Flag "Persona precisa sintetizar" Trigger + cron
Cron de síntese Persona Lê os chunks, gera Persona nova, salva pg_cron + Edge Function
Cron canal Discord Busca novas mensagens → classifica → vira fonte pg_cron + Edge Function

Por que isso não depende do OpenClaw

Todas essas rotinas rodam dentro do Supabase (pg_cron + Edge Functions Deno). OpenClaw é opcional como front de execução de agentes complexos — a alimentação e o RAG vivem em baixo dele. Se o Pedro demorar 3 meses pra liberar, o sistema segue funcionando.

Parte 7

As 5 regras de ouro pra economizar token

  1. Nunca jogar Cérebro inteiro no promptSempre RAG com top-K (8 a 12 chunks). Fim de papo.
  2. Chunks de 500 tokens com overlap de 50Recomendação OpenAI/Anthropic. Menor que isso perde contexto. Maior desperdiça.
  3. System prompt sempre primeiro pra usar cacheOpenAI e Anthropic cobram 50–90% menos em tokens cacheados. Funciona se o prefixo do prompt for igual em toda chamada. Nosso padrão: [System prompt Copy Chief] + [Chunks] + [Pergunta]. System primeiro cacheia. Troca só os chunks.
  4. Markdown, não JSON estruturadoJSON tem overhead de aspas, chaves, vírgulas. MD passa a mesma informação com 30% menos tokens.
  5. Embedding uma vez só, busca infinitaReprocessa chunk só quando a fonte muda. Busca custa microcentavos. Não tem motivo pra re-embeddar tudo periodicamente.

Fazendo as 5 certinhas: 1000 tarefas por mês no agente de Copy do Elo = ~R$ 50. Mesmo escalando pra 10 produtos × 10 agentes × 1000 tarefas = ~R$ 5 mil/mês. Caberia em qualquer orçamento da Pinguim.

Resumo executivo

O que isso significa em uma frase

Cada Cérebro é um estado vivo no Supabase. Cada agente é uma função pura que, a cada tarefa, busca os 8 trechos mais relevantes do Cérebro do produto que precisa, monta um prompt com seu system prompt especialista e os trechos, e responde. Não há re-treino. Não há drift. Não há fine-tuning. Custo por tarefa: centavos. Reprodutível pra qualquer cliente futuro que você tiver — esse é o case.
