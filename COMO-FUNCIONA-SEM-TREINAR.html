I Etapa 01 — Alimentar

Cada produto tem um Cérebro, e cada Cérebro precisa ser alimentado antes de servir.

Aulas, páginas, depoimentos, objeções, sacadas. Tudo entra no Cérebro em formato único, vai pro Supabase e é transformado em vetores buscáveis. O agente nunca conversa com o Cérebro inteiro — só com o pedacinho que importa.

A regra de ouro da alimentação

Antes de gerar qualquer coisa, o Cérebro precisa existir. Cada fonte que entra é quebrada em pedaços pequenos — cerca de 500 tokens cada — e vetorizada. Esses pedacinhos ficam guardados no Supabase, prontos pra serem consultados instantaneamente.

É isso que a gente chama de RAG — Retrieval-Augmented Generation. O agente não memoriza o Cérebro. Ele consulta o Cérebro no instante em que precisa.

500.
tokens por chunk, o tamanho ideal para preservar contexto sem desperdiçar capacidade. recomendação Anthropic + OpenAI
II O que o RAG muda

RAG é o coração da redução de custo.

Imagine o Cérebro Elo maduro: 100 fontes grandes — aulas transcritas, páginas, depoimentos, sacadas. Juntas, elas formam centenas de milhares de tokens. Se o agente recebesse tudo isso em toda chamada, a conta explodiria.

Sem RAG

US$ 1,00

Cérebro inteiro enviado em toda chamada. ~400 mil tokens por requisição. Insustentável em qualquer escala.

Com RAG

US$ 0,02

Só os 8 trechos mais relevantes viajam no prompt. ~8 mil tokens. Barato o bastante pra rodar em qualquer volume.

50 vezes mais barato. E a qualidade não cai — na verdade, melhora, porque o agente recebe só contexto relevante, sem ruído.

Como RAG opera, passo a passo

01
Quebra em pedaços
Cada fonte vira N chunks de ~500 tokens · 1 aula de 2h ≈ 40 chunks
02
Vetoriza pela OpenAI
text-embedding-3-small · custo: US$ 0,00002 por chunk · irrelevante
03
Guarda no Supabase
pgvector nativo · zero Pinecone, zero Weaviate, zero serviço extra
04
Agente consulta on-demand
pergunta vira vetor · Supabase devolve top-8 chunks mais próximos · esses vão pro prompt
III Portas de entrada

Três caminhos, um mesmo destino.

Toda fonte vira vetor antes de entrar no Cérebro. Mas quem faz essa vetorização depende de de onde a fonte vem. A gente trabalha com três portas diferentes — todas acabam no mesmo lugar, no mesmo formato.

I. Porta Um · Carga Inicial

Do que já temos hoje

Transcrições no VS Code, PDFs, briefings. Tudo isso entra via script rodando na máquina do André, uma única vez, com a chave da OpenAI ficando segura em um .env local — nunca no GitHub.

// rodado 1x só, da sua máquina node scripts/import-elo-local.mjs → lê 21 aulas do Cérebro Elo → 842 chunks · custo ~R$ 0,09 → populado no Supabase
II. Porta Dois · Upload Manual

Quando alguém sobe arquivo pelo painel

O usuário arrasta um PDF no Mission Control. Uma Edge Function do Supabase recebe o arquivo, extrai, chunka, vetoriza e grava. A chave da OpenAI fica guardada como secret no servidor — nunca trafega pelo navegador.

// disparado pelo usuário no painel Browser → Supabase Storage → Edge Function → PDF vira Markdown limpo → chunka + vetoriza → INSERT em cerebro_fontes + chunks
III. Porta Três · Cron Automático

Quando o sistema alimenta sozinho

Você configura no painel: "a cada hora, puxa depoimentos novos do Discord". O pg_cron (cron nativo do Postgres) dispara uma Edge Function igual à da Porta Dois. Roda sem ninguém apertar botão.

// configurado pelo painel, executado pelo Supabase pg_cron → a cada 1h → busca no Discord mensagens novas → classifica tipo (depoimento? sacada?) → chunka + vetoriza → INSERT no Cérebro correto

Não importa de qual porta a fonte veio — ela acaba no mesmo Supabase, no mesmo formato. A partir daí, todo agente consulta igual, sem saber de onde veio.

Escala do Pinguim hoje

Se a gente imagina 1000 tarefas por mês no agente de Copy do Elo — seja página, anúncio, e-mail — a conta fecha em ~R$ 50 por agente, por mês.

Extrapolando pra toda a operação:

R$ 5k
Mensais pra rodar 10 produtos × 10 agentes × 1000 tarefas. Estimativa conservadora · a conta real tende a ser menor

E cada tarefa deixa rastro. Toda execução grava qual pergunta foi feita, quais chunks o agente usou, qual saída gerou. Virou audit trail completo da Qualidade — em qualquer reclamação, a gente sabe o que alimentou a resposta.

V Regras de ouro

Cinco disciplinas pra manter a conta baixa.

Nunca joga o Cérebro inteiro no prompt

Sempre RAG com top-K — 8 a 12 chunks por chamada. Fim de papo. Quem joga tudo paga 50 vezes mais, sem ganhar em qualidade.

Chunks de 500 tokens, com 50 de overlap

Padrão recomendado pela OpenAI e Anthropic. Menor perde contexto, maior desperdiça. O overlap garante continuidade entre pedaços.

System prompt sempre antes — pra cache

OpenAI cobra até 90% menos em tokens cacheados. Se o prefixo do prompt é igual em toda chamada (o system prompt do Copy Chief), ele cacheia. Por isso a ordem importa: system antes, chunks depois, pergunta no fim.

Markdown sempre, nunca JSON pesado

JSON tem overhead de aspas, chaves, vírgulas. Markdown carrega a mesma informação com ~30% menos tokens. O agente entende os dois igual.

Vetoriza uma vez só, consulta infinita

Cada chunk é embeddado só quando a fonte muda. Busca é grátis (pgvector é nativo). Nunca existe razão pra re-embeddar tudo periodicamente.

O sistema em uma frase

Cada Cérebro é um estado vivo no banco.
Cada agente, uma função pura que consulta.

Nada de re-treino. Nada de drift. Nada de fine-tuning. Só contexto certo, no momento certo, a custo de centavos — replicável pra qualquer cliente futuro.
